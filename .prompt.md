# Frontend & UI/UX Development Prompt

## Purpose
This prompt is a comprehensive guide for building, reviewing, debugging, and optimizing Frontend & UI/UX code in the PFRE-Omni Laravel 10 enterprise real-estate platform. It works for all team members and supports both fixed contexts and custom arguments.

## Technology Stack
- **Templating**: Laravel Blade (server-side) + Alpine.js 3.13 (lightweight interactivity) + Vue 5 (complex components)
- **Styling**: Tailwind CSS 3.4 (utility-first) with design tokens (colors, spacing, semantic variables)
- **Libraries**: Chart.js (analytics), DataTables (tables), FullCalendar (scheduling), Leaflet (maps), Select2 (dropdowns), Flatpickr (date pickers), SortableJS (drag-drop)
- **Build Tool**: Vite 5 with laravel-vite-plugin (hot reload for development)

## Project Structure
```
resources/views/
  ├── layouts/
  │   ├── app.blade.php (root layout with Alpine store)
  │   └── partials/ (sidebar, topnav, footer)
  ├── components/ (reusable Blade components: kpi-card, data-table, alert)
  ├── crm/, hrms/, leads/, properties/ (domain-specific views)
resources/css/
  ├── app.css (Tailwind imports)
  ├── design-tokens.css (CSS custom properties for theme)
resources/js/
  ├── app.js (Alpine initialization, Axios setup)
  └── bootstrap.js (environment & plugins)
```

## Task Types & When to Use This Prompt

### 1. **Build New UI Components**
**When**: Creating new Blade components, adding forms, building dashboards
**Example invocation**: `Build a KPI card component for the dashboard with trend indicators`
**Key patterns to follow**:
- Use Blade's `@props(['prop' => 'default'])` syntax
- Compose with Tailwind utility classes (no custom CSS unless design tokens exist)
- Support dark mode with `:class="{ 'dark': darkMode }"` pattern
- Reference component example: [kpi-card.blade.php](resources/views/components/kpi-card.blade.php)

### 2. **Add Interactivity & State Management**
**When**: Implementing form interactions, toggling states, managing conditional rendering
**Example invocation**: `Add a multi-step form workflow to the leads module using Alpine.js`
**Key patterns to follow**:
- Use `x-data` for component-scoped state (not global unless app-wide toggle like dark mode)
- Leverage Alpine directives: `@click`, `x-show`, `x-if`, `x-model`, `x-for`
- For complex component logic, consider Vue 5 (available but not yet standard)
- Reference example: [leads/create.blade.php](resources/views/leads/create.blade.php) for multi-step pattern

### 3. **Integrate Data Visualization & Tables**
**When**: Building dashboards, table views, analytics, scheduling
**Example invocation**: `Create a data table with sorting, search, and export for leads`
**Key patterns to follow**:
- Data tables: Use DataTables.net; reference [data-table.blade.php](resources/views/components/data-table.blade.php)
- Charts: Use Chart.js for KPIs, revenue, pipeline analytics
- Calendars: Use FullCalendar for appointments, site visits, follow-ups
- Maps: Use Leaflet for property locations and geofencing

### 4. **Review & Refactor Frontend Code**
**When**: Analyzing code for performance, accessibility, maintainability, and consistency
**Example invocation**: `Review the dashboard.blade.php for Tailwind consistency and dark mode support`
**Key patterns to check**:
- Responsive design: Use Tailwind breakpoints (`sm:`, `md:`, `lg:`, `xl:`, `2xl:`)
- Dark mode: Apply `dark:` prefixed utilities; check if design tokens are leveraged
- Accessibility: Ensure semantic HTML, ARIA labels, keyboard navigation
- Performance: Minimize HTTP requests, defer non-critical scripts, optimize asset loading

### 5. **Debug Interactive Behavior**
**When**: Fixing Alpine.js state issues, resolving event handling problems, debugging conditional rendering
**Example invocation**: `Debug why the sidebar dropdown isn't toggling on click`
**Key patterns to check**:
- Alpine state binding: Verify `x-data` scope and reactivity
- Event handlers: Check `@click`, `@submit`, `@input` bindings
- Conditional logic: Ensure `x-if`, `x-show` conditions are correct
- DOM updates: Use Alpine Devtools to inspect reactive properties

### 6. **General Frontend Guidance**
**When**: Seeking advice on best practices, design decisions, or learning the stack
**Example invocation**: `What's the best way to structure a multi-step form in this project?`
**Key patterns to share**:
- Component reusability: Extract common UI patterns into Blade components
- Dark mode: Always support dark mode with Tailwind `dark:` utilities
- State management: Use Alpine for local component state; reserve Vue for complex feature clusters
- Design consistency: Reference design tokens for colors and spacing

## Quick Start Examples

### Create a new Blade component:
```blade
{{-- resources/views/components/my-card.blade.php --}}
@props(['title' => '', 'value' => ''])

<div class="bg-white dark:bg-gray-800 p-6 rounded-xl shadow-sm border-t-4 border-blue-500">
  <p class="text-gray-600 dark:text-gray-400 text-sm font-medium">{{ $title }}</p>
  <p class="text-3xl font-bold dark:text-white mt-2">{{ $value }}</p>
  {{ $slot }}
</div>
```

### Use Alpine.js for interactivity:
```blade
<div x-data="{ sidebarOpen: false }" class="flex h-screen">
  <aside :class="{ 'hidden md:block': !sidebarOpen }" class="w-64 bg-gray-900 text-white">
    <!-- sidebar content -->
  </aside>
  <button @click="sidebarOpen = !sidebarOpen" class="md:hidden">
    Toggle Sidebar
  </button>
</div>
```

### Apply dark mode:
```blade
<div class="bg-white dark:bg-gray-800">
  <h1 class="text-gray-900 dark:text-white">Dark mode enabled</h1>
</div>
```

## Notes for Team Collaboration
- **Code ownership**: Frontend changes should be reviewed via PR against style & accessibility guidelines
- **Component reusability**: Extract patterns into `resources/views/components/` for team-wide use
- **Dark mode mandatory**: All new components must support dark mode with `dark:` utilities
- **Mobile-first**: Design for mobile first, then enhance with `md:`, `lg:` breakpoints
- **Testing**: Verify new interactive components in browser before merging; consider E2E tests for critical flows
