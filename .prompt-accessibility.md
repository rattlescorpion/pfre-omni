# Accessibility Compliance Prompt

## Purpose
WCAG 2.1 Level AA guidelines specific to PFRE-Omni's forms, navigation, and enterprise UI. Use this when building new components, auditing for accessibility issues, or ensuring compliance for regulated industries.

## Why Accessibility Matters

**Legal**: Section 508 (US), Accessibility for Ontarians with Disabilities Act (AODA), EN 301 549 (EU)
**Business**: 1 in 4 adults in US have some disability; 82% of people with disabilities browse daily
**Technical**: Improves SEO, mobile UX, maintainability, and code quality

## WCAG 2.1 Level AA Standards

| Level | Coverage | Target |
|-------|----------|--------|
| **A** | Basic | Minimum compliance |
| **AA** | Standard | Business requirement ✅ |
| **AAA** | Enhanced | Extra effort, rarely required |

## Quick Accessibility Checklist

- [ ] **Keyboard Navigation**: All interactive elements accessible via Tab key
- [ ] **Color Contrast**: Text has minimum 4.5:1 ratio (WCAG AA)
- [ ] **Focus Indicators**: Clear focus ring on keyboard navigation
- [ ] **Form Labels**: Every input has associated `<label>`
- [ ] **Error Messages**: Errors announced to screen readers
- [ ] **Images**: All images have descriptive `alt` text
- [ ] **Headings**: Proper hierarchy (h1, h2, h3..., not skipped)
- [ ] **Semantic HTML**: Use `<button>`, `<nav>`, `<main>`, not `<div>` for everything
- [ ] **ARIA Labels**: When semantic HTML insufficient
- [ ] **Screen Reader**: Test with NVDA (Windows), JAWS, or VoiceOver (Mac)
- [ ] **Mobile**: Touch targets ≥ 44×44px (WCAG)
- [ ] **Reduced Motion**: Respect `prefers-reduced-motion` setting

## Semantic HTML Foundation

### Proper Structure
```blade
{{-- ✅ Good: Semantic HTML --}}
<header>
  <nav>
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/leads">Leads</a></li>
    </ul>
  </nav>
</header>

<main>
  <h1>Dashboard</h1>
  <section>
    <h2>Today's Leads</h2>
    <article>Lead details</article>
  </section>
</main>

<footer>
  <p>&copy; 2026 PFRE</p>
</footer>
```

### Document Outline
```html
<h1>Page Title</h1>           <!-- One h1 per page -->
  <h2>Section 1</h2>
    <h3>Subsection 1.1</h3>
    <h3>Subsection 1.2</h3>
  <h2>Section 2</h2>          <!-- Don't skip h2 → h4 -->
    <h3>Subsection 2.1</h3>
```

## Form Accessibility

### 1. Proper Label Association
```blade
{{-- ✅ Good: Label directly associated --}}
<label for="email">Email Address</label>
<input type="email" id="email" name="email" />

{{-- ❌ Bad: Label not connected --}}
<label>Email Address</label>
<input type="email" name="email" />

{{-- ✅ Also Good: Label wraps input --}}
<label>
  Email Address
  <input type="email" name="email" />
</label>
```

### 2. Required & Optional Fields
```blade
{{-- Indicate required fields --}}
<label for="name">
  Full Name
  <span aria-label="required">*</span>
</label>
<input 
  type="text" 
  id="name" 
  name="name" 
  required 
  aria-required="true"
/>

{{-- Or use text --}}
<label for="bio">Biography (Optional)</label>
<textarea id="bio" name="bio"></textarea>
```

### 3. Error Message Association
```blade
{{-- Connect error to input field --}}
<div>
  <label for="email">Email</label>
  <input 
    type="email" 
    id="email" 
    name="email"
    aria-describedby="email-error"
  />
  @error('email')
    <span id="email-error" class="text-red-600" role="alert">
      {{ $message }}
    </span>
  @enderror
</div>
```

### 4. Form Fieldsets & Legends
```blade
{{-- Group related inputs --}}
<fieldset>
  <legend>Contact Information</legend>
  
  <label for="phone">Phone Number</label>
  <input type="tel" id="phone" name="phone" />
  
  <label for="email">Email</label>
  <input type="email" id="email" name="email" />
</fieldset>

{{-- Radio button groups --}}
<fieldset>
  <legend>Property Type</legend>
  
  <label>
    <input type="radio" name="type" value="residential" />
    Residential
  </label>
  
  <label>
    <input type="radio" name="type" value="commercial" />
    Commercial
  </label>
</fieldset>
```

### 5. Multi-Step Forms
```blade
{{-- Announce current step --}}
<fieldset>
  <legend>Step 1 of 3: Personal Information</legend>
  <!-- form fields -->
</fieldset>

{{-- Or use ARIA --}}
<div 
  role="region" 
  aria-labelledby="step-heading"
  aria-live="polite"
>
  <h2 id="step-heading">Step 1 of 3: Personal Information</h2>
  <!-- form fields -->
</div>
```

## Color & Contrast

### 1. Minimum Contrast Ratios (WCAG AA)
```
Normal Text:        4.5:1 (e.g., #333 on #fff)
Large Text (18+):   3:1
UI Components:      3:1 (buttons, focus rings)
```

### 2. Check Current Colors
```blade
{{-- ✅ Good contrast: Dark text on light background --}}
<p class="text-gray-900 bg-white">
  Dark text on white (21:1 ratio)
</p>

{{-- ❌ Bad contrast: Light gray on white --}}
<p class="text-gray-300 bg-white">
  This is too light (2.5:1 - fails WCAG AA)
</p>

{{-- ✅ Good for UI: 3:1 minimum --}}
<button class="bg-blue-600 text-white">
  Blue button on white (4.5:1)
</button>
```

### 3. Test Contrast
Tools:
- **WebAIM Contrast Checker**: https://webaim.org/resources/contrastchecker/
- **Browser DevTools**: Right-click element → Inspect → Accessibility tab
- **Lighthouse**: Run audit (checks all color combinations)

### 4. Don't Rely on Color Alone
```blade
{{-- ❌ Bad: Only color indicates status --}}
<span class="text-green-500">Active</span>

{{-- ✅ Good: Color + icon + text --}}
<span class="flex items-center gap-2">
  <svg class="w-4 h-4 text-green-500">
    <circle cy="50%" cx="50%" r="50%" fill="currentColor" />
  </svg>
  <span class="text-green-700">Active</span>
</span>
```

## Keyboard Navigation

### 1. Tab Order
```blade
{{-- Tab order should follow visual order (left-to-right, top-to-bottom) --}}

{{-- ✅ Good: Natural tab order --}}
<input type="text" placeholder="First Name" />
<input type="text" placeholder="Last Name" />
<button>Submit</button>

{{-- ❌ Bad: Jumbled tab order --}}
<input type="text" placeholder="First Name" tabindex="10" />
<input type="text" placeholder="Last Name" tabindex="1" />
<button tabindex="5">Submit</button>

{{-- ⚠️ Avoid positive tabindex (use -1 to remove from flow) --}}
<button tabindex="-1">Hidden from tab order</button>
```

### 2. Focus Indicators
```css
/* Always show focus indicator */
button:focus,
input:focus,
select:focus,
textarea:focus {
  outline: 2px solid #3b82f6;
  outline-offset: 2px;
}

/* Don't remove with outline: none */
/* ❌ Bad:
button:focus {
  outline: none;  // DON'T DO THIS
}
*/
```

### 3. Skip Links (jump to main content)
```blade
{{-- Add to top of layout --}}
<a href="#main-content" class="sr-only focus:not-sr-only">
  Skip to main content
</a>

<!-- Navigation here -->

<main id="main-content">
  <!-- Page content -->
</main>
```

### 4. Keyboard-Triggered Modals
```blade
{{-- Modal should trap focus and close with Escape --}}
<div 
  x-show="modalOpen" 
  @keydown.escape="modalOpen = false"
  role="dialog"
  aria-modal="true"
  aria-labelledby="modal-title"
>
  <h2 id="modal-title">Confirm Action</h2>
  <button @click="modalOpen = false">Close</button>
  <button @click="confirmAction()">Confirm</button>
</div>
```

## ARIA (Accessible Rich Internet Applications)

### 1. ARIA Labels
```blade
{{-- When label text isn't visible --}}
<button 
  aria-label="Close modal"
  class="text-2xl leading-none"
>
  ×
</button>

{{-- When label element exists but needs additional context --}}
<button 
  aria-label="Delete lead #1234"
>
  Delete
</button>
```

### 2. ARIA Descriptions
```blade
{{-- Provide additional context --}}
<input 
  type="password" 
  aria-describedby="password-hint"
/>
<p id="password-hint" class="text-sm text-gray-600">
  Must be at least 8 characters with a number and symbol.
</p>
```

### 3. ARIA Live Regions (announce dynamic content)
```blade
{{-- Announce updated content to screen readers --}}
<div 
  aria-live="polite"
  aria-atomic="true"
  id="status-message"
>
  <!-- Status will be announced -->
</div>

<script>
  // When status changes:
  document.getElementById('status-message').textContent = 'Lead created successfully!';
</script>

{{-- polite: Wait for pause in speech (default) --}}
{{-- assertive: Interrupt current speech --}}
{{-- off: Don't announce --}}
```

### 4. ARIA Roles
```blade
{{-- Use semantic HTML when possible, ARIA as fallback --}}

{{-- ✅ Good: Semantic --}}
<button>Save</button>

{{-- ⚠️ If must use div --}}
<div role="button" @click="save()" tabindex="0" @keydown.enter="save()">
  Save
</div>

{{-- Common roles --}}
<div role="navigation"><!-- nav links --></div>
<div role="main"><!-- main content --></div>
<div role="complementary"><!-- sidebar --></div>
<div role="contentinfo"><!-- footer --></div>
```

### 5. ARIA States & Properties
```blade
{{-- Indicate state --}}
<button 
  aria-pressed="false"
  @click="this['aria-pressed'] = this['aria-pressed'] === 'false' ? 'true' : 'false'"
>
  Toggle Feature
</button>

{{-- Indicate expanded/collapsed --}}
<button 
  aria-expanded="false"
  @click="open = !open"
>
  Show Details
</button>
<div x-show="open" :aria-hidden="!open">
  Details content...
</div>

{{-- Indicate current page --}}
<a href="/leads" aria-current="page">Leads</a>
<a href="/properties">Properties</a>

{{-- Indicate loading --}}
<div aria-busy="true" aria-label="Loading leads...">
  <!-- spinner or skeleton -->
</div>
```

## Screen Reader Testing

### 1. Windows: NVDA (Free)
- Download: https://www.nvaccess.org/
- Start: Install and press `Insert + n` to start reading
- Common commands:
  - `h` - Next heading
  - `t` - Next table
  - `f` - Next form field
  - `l` - Next list
  - `Insert + Down` - Read next line
  - `Insert + Ctrl + Home` - Read entire page

### 2. Mac: VoiceOver (Built-in)
- Start: `Cmd + F5` (or System Prefs → Accessibility)
- Commands:
  - `VO + Right Arrow` - Read next item
  - `VO + U` - Rotor (quick navigation)
  - `VO + Down` - Read entire page

### 3. JAWS (Premium, very common professionally)
- Expensive but industry standard
- Test with if possible

### 4. Test Common Scenarios
- [ ] Tab through entire page—does focus move logically?
- [ ] Read page from top—is heading structure clear?
- [ ] Find form fields—are labels read correctly?
- [ ] Activate buttons—do actions work via keyboard?
- [ ] Read links—is link text descriptive (not "click here")?
- [ ] Navigate tables—can data be understood?

## Images & Icons

### 1. Descriptive Alt Text
```blade
{{-- ✅ Good: Descriptive --}}
<img src="property.jpg" alt="Modern 3-bedroom home with pool in downtown area" />

{{-- ❌ Bad: Generic --}}
<img src="property.jpg" alt="image" />
<img src="property.jpg" alt="property.jpg" />

{{-- Decorative images (skip screen readers) --}}
<img src="divider.png" alt="" aria-hidden="true" />
```

### 2. Icon Fonts & SVGs
```blade
{{-- Icon with visible label (no alt needed) --}}
<button class="flex items-center gap-2">
  <i class="icon-search"></i>
  Search
</button>

{{-- Icon without visible label (provide aria-label) --}}
<button aria-label="Search" title="Search">
  <i class="icon-search"></i>
</button>

{{-- SVG with description --}}
<svg aria-label="Error" role="img">
  <circle cx="50" cy="50" r="40" fill="red" />
</svg>
```

### 3. Background Images
```blade
{{-- ❌ Bad: Information only in background --}}
<div style="background-image: url('chart.png')">
  <!-- no text -->
</div>

{{-- ✅ Good: Text describes image content --}}
<div 
  style="background-image: url('chart.png')"
  role="img"
  aria-label="Sales chart: 25% increase in Q1"
>
  Sales chart: 25% increase in Q1
</div>
```

## Motion & Animation

### 1. Respect Prefers-Reduced-Motion
```css
/* In Tailwind: use motion-safe and motion-reduce utilities */

@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

```blade
{{-- Disable Alpine.js animations for users with reduced motion --}}
<div 
  x-data="{ show: false }"
  x-show="show"
  @click="show = !show"
  :class="{ 'transition-all duration-300': !prefersReducedMotion }"
>
  Fade in/out
</div>
```

### 2. Animated GIFs & Videos
```blade
{{-- Provide pause control for animated content --}}
<video autoplay muted loop aria-label="Product demo">
  <source src="demo.mp4" type="video/mp4" />
  Your browser does not support the video tag.
</video>

{{-- Or pause by default --}}
<video controls aria-label="Tutorial">
  <source src="tutorial.mp4" type="video/mp4" />
</video>
```

## Data Tables

### 1. Accessible Table Structure
```blade

  <thead>
    <tr>
      <th scope="col">Name</th>
      <th scope="col">Email</th>
      <th scope="col">Status</th>
      <th scope="col">Actions</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>John Doe</td>
      <td>john@example.com</td>
      <td>Active</td>
      <td>
        <button aria-label="Edit John Doe">Edit</button>
      </td>
    </tr>
  </tbody>
</table>
```

### 2. Complex Tables with Headers
```blade
{{-- Table with multiple header levels --}}
<table>
  <thead>

    </tr>
    <tr>
      <th scope="col"></th>
      <th scope="col">Jan</th>
      <th scope="col">Feb</th>
      <th scope="col">Mar</th>
    </tr>
  </thead>

    <tr>
      <td>North</td>

    </tr>
  </tbody>
```

## Mobile Accessibility

### 1. Touch Targets
```css
/* Minimum 44×44px (WCAG) */
button,
a,
input[type="checkbox"],
input[type="radio"],
input[type="button"]
{
  min-height: 44px;
  min-width: 44px;
  padding: 12px;
}

/* Avoid small tap targets */
/* ❌ Bad: 20×20px icon */
/* ✅ Good: 44×44px touchable area */
```

### 2. Responsive Text Sizing
```blade
{{-- Zoom to at least 200% should remain readable --}}
<p class="text-base md:text-lg lg:text-xl">
  Scalable text content
</p>

{{-- Don't disable zoom --}}
{{-- ❌ Bad: <meta name="viewport" content="user-scalable=no"> --}}

{{-- ✅ Good: --}}
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

### 3. Touch-Friendly Spacing
```blade
{{-- Form inputs need spacing on mobile --}}
<div class="space-y-4 md:space-y-6">
  <input type="text" class="w-full px-4 py-3" placeholder="Name" />
  <input type="email" class="w-full px-4 py-3" placeholder="Email" />
  <button class="w-full py-3">Submit</button>
</div>
```

## Accessibility Audit Checklist

### Automated Tools (find ~25% of issues)
- [ ] Run Chrome DevTools Lighthouse → Accessibility audit
- [ ] Use WAVE browser extension
- [ ] Run Axe DevTools (https://www.deque.com/axe/devtools/)
- [ ] Check with color blindness simulator

### Manual Testing (find ~75% of issues)
- [ ] Keyboard navigation: Tab through page, use Enter/Space
- [ ] Screen reader: Test with NVDA or VoiceOver
- [ ] Color contrast: Check with WebAIM tool
- [ ] Zoom: Test at 200% zoom level
- [ ] Mobile: Test on touch device or emulator
- [ ] Form usability: Fill out forms without mouse

### Testing Checklist Template
```markdown
## Accessibility Audit: [Component/Page Name]

### Keyboard Navigation
- [ ] All interactive elements reachable by Tab
- [ ] Tab order logical (left-to-right, top-to-bottom)
- [ ] Focus visible on all interactive elements
- [ ] Can close modals with Escape key

### Screen Reader
- [ ] Page structure clear (proper headings, landmarks)
- [ ] Form labels associated with inputs
- [ ] Error messages announced
- [ ] Links have descriptive text (not "click here")
- [ ] Images have alt text
- [ ] Dynamic content updates announced

### Visual
- [ ] Text contrast minimum 4.5:1
- [ ] UI component contrast minimum 3:1
- [ ] Color not sole indicator of information
- [ ] Page readable at 200% zoom
- [ ] Animations respect prefers-reduced-motion

### Mobile
- [ ] Touch targets minimum 44×44px
- [ ] Text size at least 16px for comfortable reading
- [ ] Zoom enabled (user-scalable=yes)

### Forms
- [ ] All required fields marked and indicated
- [ ] Errors associated with fields
- [ ] Help text associated with inputs
- [ ] Form can be submitted via keyboard
```

## Common Issues & Solutions

| Issue | Solution |
|-------|----------|
| **No focus ring** | Remove `outline: none` and style focus properly |
| **Missing alt text** | Add descriptive alt text to all images |
| **Poor color contrast** | Use WebAIM tool to find adequate combinations |
| **Unlinked labels** | Add `for="id"` to `<label>` matching input `id` |
| **No error indication** | Associate error message with input using `aria-describedby` |
| **Skipped heading levels** | Use h1, h2, h3 in proper sequence |
| **Links say "click here"** | Use descriptive link text ("View lead details", not "click here") |
| **Unannounced dynamic updates** | Use `aria-live="polite"` region |
| **Inaccessible modals** | Add `role="dialog"` and trap focus |
| **Unreadable at zoom** | Test at 200% zoom, ensure layout responsive |
